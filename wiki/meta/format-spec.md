**Version:** 2.0

## 1. Scope

This spec defines the **structural format** for:
- `wiki/concepts/<concept-slug>.md` — knowledge atoms
- `wiki/sources/src_<slug>.md` — per-source notes

**Out of scope:**
- Index files (`type: index`) → see `wiki/meta/index-spec.md`
- Raw content files (`raw/<type>/YYYY-MM-DD_<slug>.md`) → format defined in Ingest Agent skill

**What this spec covers:**
- File naming conventions
- Frontmatter schema (fields, types, order)
- Required sections and their order
- Markdown syntax rules
- Wikilink format

**What this spec does NOT cover:**
- Content quality (factual accuracy, completeness, coherence) → see Output Validator
- Vietnamese language quality (grammar, spelling) → see Output Validator
- Content depth or length requirements → see Output Validator

Files in other folders (`raw/`, `wiki/tag/`, `wiki/topic/`, `wiki/reviews/`) have different formats and are not covered here.

## 2. Concept file format

### 2.1 Naming convention

```
wiki/concepts/<concept-slug>.md
```

**Rules:**
- `<concept-slug>` must be lowercase with hyphens (e.g., `ai-security-tools`, `defi-exploits`)
- No spaces, underscores, or special characters except hyphens
- Vietnamese diacritics allowed in slug (e.g., `tối-ưu-hóa-context`)
- Max length: 50 characters

**Validation:**
- ERROR: Slug contains invalid characters (spaces, underscores, special chars)
- ERROR: Slug exceeds 50 characters
- WARNING: Slug uses uppercase letters

### 2.2 Required frontmatter

```yaml
---
type: concept
status: draft | reviewed | needs-revision
main_tag: <one-from-pool-a>
sub_tags: [<one-to-three-from-pool-b>]
topic: <lowercase-hyphen-slug>
sources:
  - [[wiki/sources/src_<slug>]]
last_updated: YYYY-MM-DD
---
```

**Field definitions:**

| Field | Type | Required | Validation |
|---|---|---|---|
| `type` | string | Yes | Must be exactly `concept` |
| `status` | enum | Yes | One of: `draft`, `reviewed`, `needs-revision` |
| `main_tag` | string | Yes | Must exist in `TAGS.md` Pool A |
| `sub_tags` | array | Yes | 1–3 tags from `TAGS.md` Pool B |
| `topic` | string | Yes | Lowercase-hyphen slug, max 60 chars |
| `sources` | array | Yes | Wikilinks to `wiki/sources/src_*.md` files |
| `last_updated` | date | Yes | ISO format `YYYY-MM-DD` |

**Field order (required):**
1. `type`
2. `status`
3. `main_tag`
4. `sub_tags`
5. `topic`
6. `sources`
7. `last_updated`

**Validation:**
- ERROR: Missing required field
- ERROR: Field has wrong type (e.g., `sub_tags` is string not array)
- ERROR: `status` not one of allowed values
- ERROR: `main_tag` not in TAGS.md Pool A
- ERROR: `sub_tags` contains tag not in TAGS.md Pool B
- ERROR: `sub_tags` has <1 or >3 items
- ERROR: `last_updated` not valid ISO date (YYYY-MM-DD)
- ERROR: Invalid YAML syntax
- WARNING: Field order does not match spec
- WARNING: `sources` array is empty

### 2.3 Required sections (in order)

```markdown
# <Concept Title>

## Definition

## Key ideas

## Related concepts

## Sources
```

**Section rules:**
- All sections required
- Section order must match above
- Use H2 (`##`) for all section headers
- `## Sources` section auto-generated by Compile Agent — manual edits will be overwritten

**Validation:**
- ERROR: Missing required section
- ERROR: Section uses wrong heading level (not H2)
- ERROR: Duplicate section header
- WARNING: Section order does not match spec
- INFO: Could add optional `## Notes` section for Julius's annotations

### 2.4 Internal links

**Rules:**
- Use wikilink syntax: `[[target]]` or `[[target|display text]]`
- Concepts link to concepts: `[[other-concept]]`
- Concepts link to sources: `[[wiki/sources/src_slug]]`
- No external URLs in concept body — put in source files instead

**Validation:**
- ERROR: Link uses markdown syntax `[text](url)` for internal wiki link
- ERROR: Wikilink has spaces around brackets: `[[ target ]]`
- WARNING: Wikilink points to non-existent file
- INFO: Could use wikilink instead of full path for internal links

## 3. Source file format

### 3.1 Naming convention

```
wiki/sources/src_<slug>.md
```

**Rules:**
- Prefix `src_` required
- `<slug>` matches the original raw file slug (e.g., `src_anthropic-claude-code`)
- Same slug rules as concepts (lowercase-hyphen, max 50 chars)

**Validation:**
- ERROR: Missing `src_` prefix
- ERROR: Slug contains invalid characters
- ERROR: Slug exceeds 50 characters
- WARNING: Slug uses uppercase letters

### 3.2 Required frontmatter

```yaml
---
type: source
original: raw/<type>/YYYY-MM-DD_<slug>.md
main_tag: <one-from-pool-a>
sub_tags: [<one-to-three-from-pool-b>]
topic: <lowercase-hyphen-slug>
date_compiled: YYYY-MM-DD
url: <optional-original-url>
author: <optional-author-name>
---
```

**Field definitions:**

| Field | Type | Required | Validation |
|---|---|---|---|
| `type` | string | Yes | Must be exactly `source` |
| `original` | path | Yes | Must point to existing file in `raw/` |
| `main_tag` | string | Yes | Must exist in `TAGS.md` Pool A |
| `sub_tags` | array | Yes | 1–3 tags from `TAGS.md` Pool B |
| `topic` | string | Yes | Lowercase-hyphen slug |
| `date_compiled` | date | Yes | ISO format `YYYY-MM-DD` |
| `url` | string | No | Original URL if available |
| `author` | string | No | Original author if known |

**Field order (required):**
1. `type`
2. `original`
3. `main_tag`
4. `sub_tags`
5. `topic`
6. `date_compiled`
7. `url` (if present)
8. `author` (if present)

**Validation:**
- ERROR: Missing required field
- ERROR: Field has wrong type
- ERROR: `original` path does not point to existing raw file
- ERROR: `main_tag` not in TAGS.md Pool A
- ERROR: `sub_tags` contains tag not in TAGS.md Pool B
- ERROR: `sub_tags` has <1 or >3 items
- ERROR: `date_compiled` not valid ISO date
- ERROR: Invalid YAML syntax
- WARNING: Field order does not match spec

### 3.3 Required sections (in order)

```markdown
# <Source Title>

## Metadata

## Summary

## Key points

## Concepts referenced
```

**Section rules:**
- All sections required
- Section order must match above
- Use H2 (`##`) for all section headers
- `## Concepts referenced` auto-updated by Compile Agent when concepts reference this source

**Validation:**
- ERROR: Missing required section
- ERROR: Section uses wrong heading level (not H2)
- ERROR: Duplicate section header
- WARNING: Section order does not match spec
- INFO: Could add optional `## Original excerpts` section

## 4. Common markdown syntax rules

### 4.1 Headers

**Rules:**
- Use ATX headers (`#`, `##`, `###`) not Setext (`===`, `---`)
- File title must be H1 (`#`)
- Section headers must be H2 (`##`)
- Subsections (if any) must be H3 (`###`)
- No skipping levels (H1 → H3 without H2)

**Validation:**
- ERROR: File has no H1 title
- ERROR: File has multiple H1 headers
- ERROR: Section uses H1 instead of H2
- ERROR: Setext headers used
- WARNING: Heading level skipped

### 4.2 Code blocks

**Rules:**
- Code blocks must specify language: ` ```yaml ` not ` ``` `
- Use lowercase language names: `python` not `Python`
- Common languages: `yaml`, `markdown`, `bash`, `python`, `javascript`, `json`

**Validation:**
- ERROR: Code block missing language tag
- WARNING: Language tag uses uppercase
- INFO: Could specify language for unlabeled code block

### 4.3 Lists

**Rules:**
- Use `-` for unordered lists (not `*` or `+`)
- Use `1.` for ordered lists
- Consistent indentation (2 or 4 spaces)
- No mixing bullet styles in same list

**Validation:**
- WARNING: Inconsistent bullet markers (mixing `-`, `*`, `+`)
- WARNING: Inconsistent indentation
- INFO: Prefer `-` for bullet lists

### 4.4 Emphasis

**Rules:**
- Use `**bold**` for bold (not `__bold__`)
- Use `*italic*` for italic (not `_italic_`)
- Use `` `code` `` for inline code

**Validation:**
- INFO: Prefer `**bold**` over `__bold__`
- INFO: Prefer `*italic*` over `_italic_`

## 5. Frontmatter syntax rules

### 5.1 YAML format

**Rules:**
- Must be valid YAML
- Must be first thing in file (no content before opening `---`)
- Opening and closing `---` on separate lines
- No trailing spaces in values
- Arrays use bracket syntax: `[item1, item2]` not YAML list syntax

**Validation:**
- ERROR: Invalid YAML syntax
- ERROR: Frontmatter not at start of file
- ERROR: Missing opening or closing `---`
- WARNING: Array uses YAML list syntax instead of brackets

### 5.2 Date format

**Rules:**
- ISO 8601 format: `YYYY-MM-DD`
- No time component
- Must be valid calendar date
- Year must be 2000-2030 (reasonable range)

**Validation:**
- ERROR: Invalid date format (not YYYY-MM-DD)
- ERROR: Invalid calendar date (e.g., 2026-13-45)
- WARNING: Date outside reasonable range (before 2000 or after 2030)

### 5.3 Tag format

**Rules:**
- Tags in frontmatter do NOT use `#` prefix
- Tags must be lowercase-hyphen format
- Tags must exist in TAGS.md

**Validation:**
- ERROR: Tag uses `#` prefix in frontmatter
- ERROR: Tag contains invalid characters (spaces, underscores)
- ERROR: Tag not found in TAGS.md
- WARNING: Tag uses uppercase letters

## 6. Wikilink rules

### 6.1 Syntax

**Rules:**
- Must use double brackets: `[[target]]`
- Optional display text: `[[target|display text]]`
- No spaces around brackets: `[[target]]` not `[[ target ]]`
- **Use bare slugs** (Obsidian resolves automatically):
  - Concepts: `[[concept-slug]]` not `[[wiki/concepts/concept-slug]]`
  - Sources: `[[src_slug]]` not `[[wiki/sources/src_slug]]`
- Full paths allowed but discouraged (verbose, breaks on folder rename)
- No wikilinks to files outside `wiki/`

**Examples:**
```
✅ Correct (bare slug):
[[claude-code-workflow]]
[[src_anthropic-claude-code]]

⚠️ Acceptable (full path):
[[wiki/concepts/claude-code-workflow]]
[[wiki/sources/src_anthropic-claude-code]]

❌ Wrong (external):
[[../raw/articles/file.md]]
```

**Validation:**
- ERROR: Wikilink has spaces around brackets
- ERROR: Wikilink points to file outside `wiki/`
- WARNING: Wikilink points to non-existent file
- INFO: Could use shorter wikilink (omit `wiki/` prefix)

### 6.2 Link targets

**Rules:**
- Concepts link to concepts: `[[concept-slug]]`
- Concepts link to sources: `[[src_slug]]`
- Sources link to concepts: `[[concept-slug]]`
- No external URLs in wikilinks — use markdown links for external

**Validation:**
- ERROR: Wikilink to external URL (use markdown link instead)
- WARNING: Broken wikilink (target file does not exist)

## 7. Validation severity levels

When Format Validator finds issues, it assigns severity:

| Severity | Meaning | Action |
|---|---|---|
| **ERROR** | Breaks format spec, blocks compilation | Must fix before file is usable |
| **WARNING** | Violates convention, should fix | File usable but needs improvement |
| **INFO** | Style suggestion, optional | Nice to fix, not required |

**Escalation:**
- 3+ ERRORs in one file → Consider moving to `wiki/drafts/`
- 5+ WARNINGs in one file → Flag for systematic issue
- 10+ INFOs in one file → Likely over-reporting, review criteria

```markdown
sub_tags: [tools, automation]

  - [[wiki/sources/src_anthropic-claude-code]]

# Claude Code Workflow

## Definition

Claude Code is an AI-assisted development workflow where Claude autonomously implements features given high-level specifications. The developer focuses on system design and decision-making rather than writing code.

## Key ideas

- Claude can read, write, and execute code autonomously
- Developer role shifts from coding to architecture
- Workflow: design → spec → Claude implements → review → iterate
- Best for greenfield projects and well-defined tasks
- Requires clear specifications for accurate implementation

## Related concepts

- [[vibe-coding]]
- [[ai-assisted-development]]
- [[spec-driven-development]]

## Sources

- [[wiki/sources/src_anthropic-claude-code]]
```

## 10. Validation workflow

When Format Validator runs:

1. **Scan** `wiki/concepts/` and `wiki/sources/`
2. **For each file:**
   - Parse frontmatter (YAML validation)
   - Check required fields exist and have correct type
   - Verify field order matches spec
   - Verify tags exist in `TAGS.md`
   - Check section structure matches spec
   - Validate wikilinks use correct syntax
   - Check markdown syntax (headers, code blocks, lists)
3. **Generate report:** `wiki/reviews/YYYY-MM-DD_format-report.md`
4. **Update:** `wiki/reviews/_action-required.md`

**Format Validator never modifies files** — only reports issues.

**Content quality checks** (factual accuracy, completeness, coherence, Vietnamese quality) are handled by Output Validator, not Format Validator.
